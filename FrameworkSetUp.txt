
**Framework Architecture (How It Is Implemented)**

1. Test Runner Layer  
- Playwright test runner is configured with:
  - test directory and matching
  - retries = 3
  - workers = 1 (sequential execution)
  - reporters: list + allure + html
  - trace/video/screenshot on failure
- This favors reliability and debuggability over speed.

Evidence:
- playwright.config.ts

2. Suite Orchestration Layer  
- Global setup and teardown are used to clean and publish Allure artifacts.
- There is also script-level cleanup in npm pretest, so cleanup happens in two places.

Evidence:
- globalSetup.ts
- globalTeardown.ts
- package.json

3. Fixture and Session Layer  
- Custom fixtures extend Playwright test object:
  - authenticatedPage
  - approverAuthenticatedPage
  - loginPage
- Shared login flow is centralized in performLogin, reducing duplication.

Evidence:
- baseFixtures.ts

4. Page Abstraction Layer  
- BasePage is a common superclass containing:
  - safeClick fallback logic
  - page readiness checks
  - dialog helpers
  - grid row search + polling helpers
  - dropdown selection by label text
- Business pages inherit from BasePage and only implement domain selectors/actions.

Evidence:
- BasePage.ts
- LoginPage.ts
- ApprovalLevelMatrixPage.ts
- ApprovalMatrixPage.ts

5. Test Scenario Layer  
- Tests act as orchestration scripts:
  - navigate
  - call page methods
  - branch based on config flags
  - validate outcomes
  - log each step
- Examples include CRUD matrix tests and role-based workflow tests.

Evidence:
- TC001_LoginTest.spec.ts
- TC002_ApprovalLevelMatrixTest.spec.ts
- TC005_ApprovalMatrixTest.spec.ts
- TC026_RequestCreationTest.spec.ts
- TC029_ApproverReviewTest.spec.ts

6. Data and Environment Layer  
- Massive config object loads all test input from environment.
- Naming convention encodes lifecycle:
  - create values
  - before-edit selectors
  - edit values
  - delete selectors
- Mostly environment-driven; file-driven data appears in upload scenario.

Evidence:
- config.ts
- MassRequestUpload.xlsx
- TC021_MassRequestUploadTest.spec.ts

7. Logging and Diagnostics Layer  
- Winston logger with label-based cached instances.
- Logs to both console and rotating file.
- Structured messages improve failure triage.

Evidence:
- Logger.ts

**Design Patterns Used (Interview-Focused)**

1. Page Object Model (Primary Pattern)  
- Each screen is encapsulated in a dedicated page class.
- Selectors and actions are hidden from tests.
- Benefit: changes in UI locators mostly affect one class.

Evidence:
- LoginPage.ts
- RequestCreationPage.ts
- SubmitterReviewPage.ts

2. Inheritance for Shared Behavior (Base Class Pattern)  
- BasePage provides generalized mechanisms used by all pages.
- Reduces duplicated resilience logic.

Evidence:
- BasePage.ts

3. Fixture-based Dependency Injection  
- Test dependencies are injected per test through fixture extension.
- Supports role-specific authentication and reusable page provisioning.

Evidence:
- baseFixtures.ts

4. Template/Workflow Encapsulation (Partial)  
- Shared login workflow is centralized in performLogin.
- Tests call fixtures, not low-level login steps.

Evidence:
- baseFixtures.ts

5. Strategy via Configuration  
- Behavior switches based on environment values:
  - request type
  - tabs
  - action case values
- Same test logic supports different business paths.

Evidence:
- config.ts
- TC027_SubmitterReviewTest.spec.ts
- TC028_AdminReviewTest.spec.ts
- TC029_ApproverReviewTest.spec.ts

6. Flyweight-like Logger Cache  
- Logger instances are cached by label.
- Prevents recreation and keeps consistent context tags.

Evidence:
- Logger.ts

**Execution Flow (End-to-End Mental Model)**

1. Run starts.
2. Global setup removes prior report artifacts.
3. Playwright applies project config, timeouts, retries, reporters.
4. Fixture logs in user or approver and returns ready page.
5. Test instantiates relevant page object.
6. Test performs scenario actions.
7. BasePage utilities absorb flaky behavior with waits/fallbacks.
8. Assertion validates state.
9. Artifacts are collected (trace/video/screenshot on failure).
10. Global teardown generates and opens Allure report.

**Tricky Interview Questions You Are Likely to Get (With Strong Answer Direction)**

1. Why POM instead of direct selectors in tests?  
- Separation of concerns, maintainability, locator change impact isolation, reusable domain methods.

2. Why a BasePage?  
- Centralized reliability logic (safeClick, waitForPageReady, dialog/grid helpers), consistency across pages.

3. Why fixtures instead of beforeEach hooks in every file?  
- Dependency injection, composability, role-specific sessions, reduced duplication.

4. Why workers = 1 when Playwright supports parallelism?  
- System under test has shared-state dependencies; deterministic sequencing preferred for stability. Can scale with test-data isolation later.

5. How do you handle flaky Mendix controls?  
- visible scoping, title-attribute reads, cell-level clicks, polling loops, controlled fallbacks.

6. Why retries = 3?  
- To absorb transient environment/network/UI jitter while still retaining failure artifacts for root-cause analysis.

7. How do you prove this is not just script automation?  
- Show layered architecture: runner, fixtures, base abstractions, page objects, config strategy, reporting pipeline.

8. How do you manage role-based tests?  
- Dedicated approverAuthenticatedPage fixture with shared login helper and independent credential source.

9. How do you keep tests data-driven?  
- Environment-based config keys for create/edit/delete permutations and case branching.

10. How do you balance robustness and correctness when using force click/JS fallback?  
- Use as final fallback only; primary path remains regular user-like interaction.

11. What is your assertion strategy?  
- Prefer state verification after actions; identify and improve tests currently missing terminal assertions.

12. How do you debug failures quickly?  
- Structured step logs + Playwright trace/video/screenshot + Allure scenario history.

13. How do you prevent selector brittleness?  
- Encapsulate selectors in POM and scope by visible/context-aware locators.

14. Why not use API login storage state fully?  
- Utility exists but not yet integrated broadly; UI login currently validates real login path too. Could optimize later.

Evidence:
- apiLogin.ts

15. What technical debt would you address first?  
- Replace static waits with condition waits, add missing hard assertions, refactor config into typed domain modules, de-duplicate Allure lifecycle handling.

NPM Commands – Creating the Framework from Scratch
----------------------------------------------------

Step 1: Initialize the project
  npm init -y
    - Creates package.json with default values.
    - Use npm init (without -y) to answer prompts manually.

Step 2: Install Playwright (with test runner)
  npm install --save-dev @playwright/test
    - Installs Playwright and the @playwright/test runner as a dev dependency.

  npx playwright install
    - Downloads browser binaries (Chromium, Firefox, WebKit).

  npx playwright install --with-deps
    - Also installs OS-level dependencies needed by browsers (recommended on CI/Linux).

Step 3: Install TypeScript support
  npm install --save-dev typescript ts-node @types/node
    - typescript : the TypeScript compiler.
    - ts-node    : run .ts files directly without a separate compile step.
    - @types/node: Node.js type definitions.

  npx tsc --init
    - Generates a default tsconfig.json in the project root.
    - Key options to set inside tsconfig.json:
        "target": "ES2020"
        "module": "commonjs"
        "strict": true
        "outDir": "./dist"
        "rootDir": "./src" (or wherever your source lives)

Step 4: Install Allure reporter
  npm install --save-dev allure-playwright allure-commandline
    - allure-playwright   : Playwright reporter plugin that writes Allure-compatible results.
    - allure-commandline : CLI used in globalTeardown / npm scripts to generate the HTML report.

Step 5: Install Winston logger
  npm install winston
  npm install --save-dev @types/winston
    - winston         : structured logging library.
    - @types/winston  : TypeScript types for winston.

Step 6: Install dotenv (environment variable support)
  npm install dotenv
    - Loads .env file values into process.env at runtime.
    - Add require('dotenv').config() at the top of your config entry point.

Step 7: Install any additional utilities
  npm install --save-dev cross-env
    - Allows setting environment variables in npm scripts cross-platform (Windows + Unix).

  npm install --save-dev rimraf
    - Cross-platform rm -rf used in clean-up npm scripts.

Step 8: Verify installed packages
  npm list --depth=0
    - Lists all top-level installed packages and their versions.

Step 9: Common npm scripts to add in package.json
  "scripts": {
    "pretest"  : "rimraf allure-results test-results playwright-report",
    "test"     : "npx playwright test",
    "test:headed" : "npx playwright test --headed",
    "test:ui"  : "npx playwright test --ui",
    "report"   : "allure generate allure-results --clean -o allure-report && allure open allure-report",
    "clean"    : "rimraf allure-results allure-report test-results playwright-report"
  }

Step 10: Run tests
  npx playwright test                         -- run all tests (headless)
  npx playwright test --headed                -- run with browser visible
  npx playwright test --ui                    -- open Playwright UI mode
  npx playwright test tests/TC001_LoginTest.spec.ts  -- run a single file
  npx playwright test --grep "login"          -- run tests matching a pattern
  npx playwright test --project=chromium      -- run only on a specific browser
  npx playwright test --retries=2             -- override retry count from CLI

Step 11: Generate and view reports
  npx allure generate allure-results --clean -o allure-report
    - Converts raw Allure JSON results into a static HTML report.

  npx allure open allure-report
    - Opens the generated report in the default browser.

  npx playwright show-report
    - Opens the built-in Playwright HTML report (playwright-report/).

Step 12: Debug helpers
  npx playwright codegen https://your-app-url.com
    - Launches the Playwright inspector and records user actions as test code.

  npx playwright test --debug
    - Runs tests in debug/step-through mode with the Playwright inspector.

  PWDEBUG=1 npx playwright test          (Linux/Mac)
  set PWDEBUG=1 && npx playwright test   (Windows CMD)
  $env:PWDEBUG=1; npx playwright test    (Windows PowerShell)

Step 13: Update / audit packages
  npm outdated                    -- list packages with newer versions available
  npm update                      -- update all packages within semver range
  npm audit                       -- show known vulnerabilities
  npm audit fix                   -- auto-fix compatible vulnerability patches
  npm audit fix --force           -- force-fix including breaking version bumps (use carefully)

Step 14: Useful one-liners
  npx playwright --version        -- print installed Playwright version
  npx playwright test --list      -- list all discovered test cases without running them
  npx tsc --noEmit                -- type-check all TypeScript files without emitting output

import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  // Directory where test files are located
  testDir: './tests',

  // Pattern to match test files
  testMatch: '**/*.spec.ts',

  forbidOnly: !!process.env.CI,

  // Reporters: list (console), allure, built-in HTML
  reporter: [
    ['list'],
    ['allure-playwright', { outputFolder: 'allure-results' }],
    ['html', { outputFolder: 'playwright-report', open: 'never' }],

  // Shared settings applied to all projects

    baseURL: process.env.BASE_URL || 'https://your-app.example.com',
    headless: true,                  // set false for local debugging

    ignoreHTTPSErrors: true,         // useful for self-signed certs in test envs

Locator Strategy & Best Practices
-----------------------------------

Priority order (most to least resilient):

1. Role-based (most resilient – semantically meaningful)
   page.getByRole('button', { name: 'Submit' })
   page.getByRole('textbox', { name: 'Username' })
   page.getByRole('link',    { name: 'Dashboard' })

2. Label-associated inputs
   page.getByLabel('Email Address')

3. Placeholder text
   page.getByPlaceholder('Search...')

4. Visible text
   page.getByText('Approval Matrix')
   page.getByText('Approval Matrix', { exact: true })   ← exact match

5. Test ID (custom data attribute – most stable in controlled codebases)
   page.getByTestId('submit-btn')                       ← requires data-testid="submit-btn" in HTML

6. Title attribute
   page.locator('[title="Open Menu"]')                  ← used for Mendix status fields

7. CSS / XPath (last resort – brittle)
   page.locator('.mx-grid-row >> nth=2')
   page.locator('//button[contains(text(),"Save")]')

Scoping locators (reduces false matches):
   const modal = page.locator('.modal-dialog');
   await modal.getByRole('button', { name: 'Confirm' }).click();

Filtering by visibility (critical for Mendix duplicate controls):
   page.locator('.mx-textbox').filter({ hasText: 'Request ID' }).nth(0)
   page.locator('button', { has: page.locator(':visible') })

Chaining locators:
   page.locator('table').locator('tr').filter({ hasText: 'Pending' }).locator('td').nth(2)

Waiting for a locator to be ready:
   await expect(locator).toBeVisible({ timeout: 10_000 });
   await locator.waitFor({ state: 'visible' });

Common Playwright Assertions Reference
----------------------------------------

Import:
  import { expect } from '@playwright/test';

Visibility:
  await expect(locator).toBeVisible();
  await expect(locator).toBeHidden();

Text content:
  await expect(locator).toHaveText('Expected text');
  await expect(locator).toContainText('partial text');
  await expect(locator).toHaveText(/regex pattern/);

Input value:
  await expect(locator).toHaveValue('input value');
  await expect(locator).toBeEmpty();                    ← input is blank

Attribute:
  await expect(locator).toHaveAttribute('href', '/dashboard');
  await expect(locator).toHaveAttribute('class', /active/);

Count:
  await expect(locator).toHaveCount(5);

State:
  await expect(locator).toBeEnabled();
  await expect(locator).toBeDisabled();
  await expect(locator).toBeChecked();
  await expect(locator).not.toBeChecked();

URL / Title:
  await expect(page).toHaveURL('https://app.com/dashboard');
  await expect(page).toHaveURL(/\/dashboard/);
  await expect(page).toHaveTitle('Dashboard');

Soft assertions (do not stop test on failure – collect all failures):
  await expect.soft(locator).toBeVisible();
  await expect.soft(locator).toHaveText('Status: Approved');
  // test continues even if above fails; final summary shows all soft failures

Custom timeout override per assertion:
  await expect(locator).toBeVisible({ timeout: 20_000 });

Common Playwright Patterns
----------------------------

1. Handling Alerts / Dialogs
   page.on('dialog', async dialog => {
     console.log(dialog.message());
     await dialog.accept();             // or dialog.dismiss()
   });
   await page.locator('#delete-btn').click();

2. File Upload
   await page.locator('input[type="file"]').setInputFiles('./data/MassRequestUpload.xlsx');
   // For hidden file inputs use:
   const [fileChooser] = await Promise.all([
     page.waitForEvent('filechooser'),
     page.locator('#upload-trigger').click(),
   ]);
   await fileChooser.setFiles('./data/MassRequestUpload.xlsx');

3. New Tab / Popup Handling
   const [newPage] = await Promise.all([
     page.context().waitForEvent('page'),
     page.locator('#open-link').click(),
   ]);
   await newPage.waitForLoadState();
   await expect(newPage).toHaveTitle('New Tab Title');

4. iFrame Handling
   const frame = page.frameLocator('#my-iframe');
   await frame.locator('#submit').click();
   await expect(frame.locator('.success-msg')).toBeVisible();

5. Network Interception / Mocking
   // Block images to speed up tests
   await page.route('**/*.{png,jpg,jpeg,gif}', route => route.abort());

   // Mock an API response
   await page.route('**/api/status', async route => {
     await route.fulfill({
       status: 200,
       contentType: 'application/json',
       body: JSON.stringify({ status: 'Approved' }),
     });
   });

6. Waiting Strategies (prefer these over waitForTimeout)
   await page.waitForLoadState('networkidle');        ← wait until no network activity
   await page.waitForLoadState('domcontentloaded');
   await page.waitForSelector('.spinner', { state: 'hidden' });   ← wait for spinner to disappear
   await page.waitForResponse(resp => resp.url().includes('/api/save') && resp.status() === 200);
   await page.waitForFunction(() => document.title.includes('Dashboard'));

7. Screenshots (manual)
   await page.screenshot({ path: 'Screenshots/failure.png', fullPage: true });
   await locator.screenshot({ path: 'Screenshots/element.png' });

8. Keyboard & Mouse Actions
   await page.keyboard.press('Enter');
   await page.keyboard.type('Hello World');
   await page.mouse.move(100, 200);
   await page.mouse.click(100, 200);
   await locator.press('Tab');
   await locator.press('Escape');

9. Drag and Drop
   await page.dragAndDrop('#source', '#target');

10. Evaluate JavaScript in Browser Context
    const title = await page.evaluate(() => document.title);
    await page.evaluate(() => window.scrollTo(0, document.body.scrollHeight));
    const value = await locator.evaluate(el => el.getAttribute('data-status'));

11. Storage State (save/restore login session)
    // Save after login:
    await page.context().storageState({ path: 'state.json' });

    // Reuse in playwright.config.ts:
    use: { storageState: 'state.json' }

    // Or in a fixture:
    const context = await browser.newContext({ storageState: 'state.json' });

- Use npm ci instead of npm install (locked, faster, no accidental upgrades).
- Always store credentials in GitHub Secrets, never in workflow YAML.
- Use if: always() on report uploads so artifacts are preserved on failure.

- No browser is launched; tests run faster and are more stable.

Step 1: Extract the auth token
  1. Open the application in Chrome DevTools → Network tab.
  2. Perform a manual login.
  3. Look for a POST call to /api/auth/login (or similar endpoint).
  4. In the response body, locate the token field (JWT or session token).
  5. Note the request payload (username, password fields) for automation.

Step 2: Obtain token programmatically in Playwright

    const apiContext = await request.newContext({
      baseURL: process.env.BASE_URL,
      extraHTTPHeaders: { 'Content-Type': 'application/json' },
    });

    const response = await apiContext.post('/api/auth/login', {
      data: {
        username: process.env.USERNAME,
        password: process.env.PASSWORD,

    const token = body.token;           // adjust key to match actual response

    await apiContext.dispose();
  });

import { createLogger, format, transports, Logger } from 'winston';
import * as path from 'path';
import * as fs from 'fs';

const logDir = path.resolve(__dirname, '../../logs');
if (!fs.existsSync(logDir)) fs.mkdirSync(logDir, { recursive: true });

const logFormat = format.printf(({ level, message, timestamp, label }) => {
  return `${timestamp} [${label || 'TEST'}] ${level.toUpperCase()}: ${message}`;
});

const loggerCache = new Map<string, Logger>();

export function getLogger(label: string): Logger {
  if (loggerCache.has(label)) return loggerCache.get(label)!;

  const logger = createLogger({
    level: 'info',
    format: format.combine(
      format.label({ label }),
      format.timestamp({ format: 'YYYY-MM-DD HH:mm:ss' }),
      logFormat
    ),
    transports: [
      new transports.Console({
        format: format.combine(
          format.colorize(),
          format.label({ label }),
          format.timestamp({ format: 'YYYY-MM-DD HH:mm:ss' }),
          logFormat
        ),
      }),
      new transports.File({
        filename: path.join(logDir, 'test-execution.log'),
        maxsize: 5 * 1024 * 1024, // 5 MB
        maxFiles: 5,
      }),
    ],
  });

  loggerCache.set(label, logger);
  return logger;
}

import { request, APIRequestContext, BrowserContext } from '@playwright/test';
import { config } from '../config/config';

export async function apiLoginAndSaveState(storageStatePath: string): Promise<void> {
  const apiContext: APIRequestContext = await request.newContext();

  const loginResponse = await apiContext.post(config.apiLoginUrl, {
    data: {
      username:         config.username,
      usernameApprover: config.usernameApprover,
      password:         config.apiPassword,
    },
  });

  if (!loginResponse.ok()) {
    throw new Error(`Login API failed: ${loginResponse.status()} ${loginResponse.statusText()}`);
  }

  await apiContext.storageState({ path: storageStatePath });
  await apiContext.dispose();
}

export async function launchAuthenticatedContext(
  browser: any,
  storageStatePath: string
): Promise<BrowserContext> {
  return await browser.newContext({ storageState: storageStatePath });

import { Page, Locator, expect } from '@playwright/test';

export const TIMEOUTS = {
  action:       8000,
  visible:     10000,
  pageReady:   30000,
  dialog:       5000,
  pollInterval:  300,
  gridSearch:  10000,
} as const;

export class BasePage {
  protected page: Page;
  constructor(page: Page) { this.page = page; }

  // ── Click helpers ──
  protected async safeClick(locator: Locator, timeout = TIMEOUTS.action): Promise<void> {
    try {
      await locator.waitFor({ state: 'visible', timeout });
      await locator.click({ timeout });
    } catch {
      try {
        const handle = await locator.elementHandle({ timeout: 5000 });
        if (handle) {
          await this.page.evaluate((el) => {
            el.scrollIntoView({ block: 'center' });
            (el as HTMLElement).click();
          }, handle);
        }
      } catch { /* element not found */ }
    }
  }

  // ── Wait helpers ──
  protected async waitForVisible(locator: Locator, timeout = TIMEOUTS.visible) {
    await locator.waitFor({ state: 'visible', timeout });
  }
  protected async waitForHidden(locator: Locator, timeout = TIMEOUTS.visible) {
    await locator.waitFor({ state: 'hidden', timeout });
  }
  protected async waitForEnabled(locator: Locator, timeout = TIMEOUTS.visible) {
    await locator.waitFor({ state: 'visible', timeout });
    await expect(locator).toBeEnabled({ timeout });
  }

  /** Document ready + jQuery idle (Mendix-friendly). */
  async waitForPageReady(timeout = TIMEOUTS.pageReady): Promise<void> {
    await this.page.waitForFunction(
      () => {
        const jqActive = (window as any).jQuery ? (window as any).jQuery.active : 0;
        return document.readyState === 'complete' && jqActive === 0;
      },
      { timeout, polling: 500 }
    );
  }

  async waitUntilTitleIs(expectedTitle: string, timeout = 15000): Promise<void> {
    await this.page.waitForFunction(
      (title: string) => document.title === title,
      expectedTitle,
      { timeout, polling: 500 }
    );
  }

  protected async waitFor(ms: number) { await this.page.waitForTimeout(ms); }

  // ── Mendix dialog helpers ──
  private get mxDialogContainer()    { return this.page.locator('.mx-dialog'); }
  private get mxDialogMessage()      { return this.page.locator('.mx-dialog .modal-body p'); }
  private get mxDialogOkButton()     { return this.page.locator('.mx-dialog .mx-dialog-footer .btn.btn-primary'); }
  private get mxDialogCloseButton()  { return this.page.locator('.mx-dialog .mx-dialog-close'); }

  async isDialogVisible(timeout = TIMEOUTS.dialog): Promise<boolean> {
    try { await this.mxDialogContainer.first().waitFor({ state: 'visible', timeout }); return true; }
    catch { return false; }
  }
  async isDialogPresentQuick(): Promise<boolean> {
    return this.mxDialogContainer.first().isVisible().catch(() => false);
  }
  async getDialogMessage(): Promise<string | null> {
    try { return await this.mxDialogMessage.first().textContent(); } catch { return null; }
  }
  async closeDialog(): Promise<void> {
    try { await this.safeClick(this.mxDialogOkButton.first()); }
    catch { try { await this.safeClick(this.mxDialogCloseButton.first()); } catch { /* none */ } }
  }

  // ── Mendix grid helpers ──
  protected async findGridRow(
    columnSelectors: string[],
    matchValues: (string | null)[],
    exact = false
  ): Promise<number | null> {
    const columns = columnSelectors.map(s => this.page.locator(s));
    const counts  = await Promise.all(columns.map(c => c.count()));
    const rowCount = Math.min(...counts);
    if (rowCount === 0) return null;

    for (let i = 0; i < rowCount; i++) {
      let allMatch = true;
      for (let col = 0; col < columns.length; col++) {
        const expected = matchValues[col];
        if (!expected) continue;
        const cellText = ((await columns[col].nth(i).textContent()) || '').trim();
        const ok = exact
          ? cellText.toLowerCase() === expected.trim().toLowerCase()
          : cellText.toLowerCase().includes(expected.trim().toLowerCase());
        if (!ok) { allMatch = false; break; }
      }
      if (allMatch) return i;
    }
    return null;
  }

  protected async findGridRowWithWait(
    columnSelectors: string[],
    matchValues: (string | null)[],
    exact = false,
    timeout = TIMEOUTS.gridSearch
  ): Promise<number | null> {
    const deadline = Date.now() + timeout;
    while (Date.now() < deadline) {
      const idx = await this.findGridRow(columnSelectors, matchValues, exact);
      if (idx !== null) return idx;
      await this.waitFor(TIMEOUTS.pollInterval);
    }
    return null;
  }

  protected async selectDropdownOption(dropdown: Locator, text: string): Promise<void> {
    await dropdown.waitFor({ state: 'visible', timeout: TIMEOUTS.action });
    const options = dropdown.locator('option');
    const count = await options.count();
    for (let i = 0; i < count; i++) {
      const t = await options.nth(i).textContent();
      if (t && t.trim().toLowerCase() === text.trim().toLowerCase()) {
        await dropdown.selectOption({ label: t.trim() });
        return;
      }
    }
    throw new Error(`Dropdown option not found: "${text}"`);
  }

  protected async selectGridItemByName(cells: Locator, name: string, timeout = TIMEOUTS.visible): Promise<boolean> {
    if (!name?.trim()) return false;
    const target = name.trim().toLowerCase();
    try { await cells.first().waitFor({ state: 'visible', timeout }); } catch { return false; }
    const count = await cells.count();
    for (let i = 0; i < count; i++) {
      const text = await cells.nth(i).textContent();
      if (text && text.trim().toLowerCase() === target) {
        await cells.nth(i).click();
        return true;
      }
    }
    return false;
  }

  isYes(val: string | null | undefined): boolean {
    return (val || '').trim().toLowerCase() === 'yes';
  }
}

Section E. Mendix-Specific Engineering Rules (mandatory)
--------------------------------------------------------
1. Always scope locators to :visible because Mendix renders hidden duplicate
   tables/buttons across tabs (`dojoattachpoint='gridTable'` repeats per tab).
2. To select a row, click the <td> cell, NOT the <tr> row (Mendix often
   ignores row clicks).
3. Status text frequently lives in the title attribute, not visible text:
     await locator.getAttribute('title')
4. After every state-changing action call `waitForPageReady()` (DOM ready +
   jQuery idle) before reading the next element.
5. `safeClick` uses a JS-click fallback. The fallback can click hidden
   elements — combine with visibility scoping:
     page.locator("button:visible:text-is('Save')")
6. Dialogs: use `isDialogPresentQuick()` for non-blocking checks and
   `closeDialog()` to dismiss generic Mendix info/confirm dialogs.
7. Reference selector popups need polling: use `findGridRowWithWait()`.
