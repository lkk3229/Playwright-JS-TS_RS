# ✅ 1. Playwright Framework Structure & Why

## 📁 Framework Structure (Linear / Layered)

```
tests/           → Test cases  
pages/           → Page Object Model (POM)  
fixtures/        → Custom fixtures  
utils/           → Reusable functions (API, data, logger)  
config/          → Environment configs  
test-data/       → Static JSON data  
playwright.config.ts  
```

***

## ✅ Why I Prefer This Structure

### 🔹 1. Separation of Concerns

* Tests → business logic
* Pages → UI interactions
* Utils → reusable logic

👉 Easier debugging & maintenance

***

### 🔹 2. Scalability

* Supports 100+ / 1000+ tests
* Easy to add new modules

***

### 🔹 3. Reusability

* Page methods reused
* API helpers reused

***

### 🔹 4. Maintainability

* Locator change → update in one file

***

### 🔹 5. Team Collaboration

* QA → tests
* Automation → framework
* Developers → APIs

***

✅ **Why not flat structure?**

* Causes duplication
* Hard to scale

👉 This layered structure is **enterprise-ready**

# ✅ 2. Custom Type vs Interface (Which is More Beneficial?)

## ✅ Answer: **Interface is more beneficial (in most automation cases)**

***

## 🔹 Why Interface?

### ✅ 1. Strong Data Contracts

```ts
interface User {
  username: string;
  password: string;
}
```

👉 Ensures correct test data

***

### ✅ 2. Best for API & POM

* API response validation
* Page method inputs

***

### ✅ 3. Extendable

```ts
interface Admin extends User {
  role: string;
}
```

***

## 🔹 When to Use Custom Types

```ts
type Status = 'success' | 'fail';
```

👉 Good for:

* Enums
* Flexible values

***

## 🎯 Final Answer

👉 **Interfaces = structure & safety (most used)**  
👉 **Types = flexibility (limited use)**

# ✅ 3. How Docker is Used in Automation

***

## ✅ What Docker Does

Docker packages:

* Code
* Dependencies
* Browsers
* OS

👉 Runs tests in a **consistent environment**

***

## 🔹 Implementation

### ✅ Dockerfile

```dockerfile
FROM mcr.microsoft.com/playwright:v1.44.0

WORKDIR /app
COPY . .
RUN npm install
CMD ["npx", "playwright", "test"]
```

***

### ✅ Build & Run

```bash
docker build -t playwright-tests .
docker run playwright-tests
```

***

## 🔹 Benefits

### ✅ 1. No Environment Issues

Same setup everywhere

### ✅ 2. CI/CD Ready

* Jenkins / Azure can run directly

### ✅ 3. Parallel Execution

Multiple containers → faster tests

### ✅ 4. Isolation

Fresh environment every run

### ✅ 5. Easy Setup

No dependency installation

***

## 🎯 Final Answer

👉 Docker ensures **consistency + scalability + stability in automation**

# ✅ 4. Running Playwright in Jenkins Pipeline

***

## ✅ Jenkins Pipeline Flow

```
Code → Build → Install → Run Tests → Publish Report
```

***

## ✅ Jenkinsfile Example

```groovy
pipeline {
  agent any

  environment {
    NODE_ENV = 'test'
  }

  stages {

    stage('Checkout Code') {
      steps {
        git 'https://github.com/your-repo.git'
      }
    }

    stage('Install Dependencies') {
      steps {
        sh 'npm install'
        sh 'npx playwright install'
      }
    }

    stage('Run Tests') {
      steps {
        sh 'npx playwright test'
      }
    }

    stage('Publish Report') {
      steps {
        archiveArtifacts artifacts: 'playwright-report/**'
      }
    }
  }
}
```

***

## ✅ Advanced Features

### 🔹 Run Smoke First

```bash
npx playwright test --grep @smoke
```

***

### 🔹 Parallel Execution

```bash
npx playwright test --workers=4
```

***

### 🔹 Docker in Jenkins

```groovy
sh 'docker run playwright-tests'
```

***

## 🎯 Final Answer

👉 Jenkins automates **build validation, execution, and reporting**

# ✅ 5. Automate API with Authentication

***

## ✅ Step 1: Login API

```ts
const res = await request.post('/api/login', {
  data: {
    username: process.env.USERNAME,
    password: process.env.PASSWORD
  }
});
```

***

## ✅ Step 2: Extract Token

```ts
const { token } = await res.json();
```

***

## ✅ Step 3: Use Token

```ts
const response = await request.get('/api/orders', {
  headers: {
    Authorization: `Bearer ${token}`
  }
});
```

***

## ✅ Step 4: Reusable Helper

```ts
export async function getToken(request) {
  const res = await request.post('/api/login', { data: { /* credentials */ } });
  return (await res.json()).token;
}
```

***

## ✅ Step 5: Fixture (Best Practice)

```ts
test.extend({
  token: async ({ request }, use) => {
    const token = await getToken(request);
    await use(token);
  }
});
```

***

## 🎯 Final Answer

👉 **Login → Get token → Reuse token → Validate responses**
